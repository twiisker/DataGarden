Softwaresystem

DEF.: Ein Softwaresystem ist eine Menge von Programmen zusammen mit denen sie begleitenden Artefakten, die für ihre Nutzung notwendig oder hilfreich sind. Derartige Artefakte sind z.B.: Anforderungsbeschreibung, Entwürfe, Testsuites, Benutzungshandbücher, Konfigurationsdateien, Skripte, …

Objekttechnologie

Wir arbeiten mit Objekten $\rightarrow$ Programmierparadigma: Objektorientiertes Programmieren (OOP) Sprachen, die nicht OO sind, bezeichnet man als „prozedural".

Allgemeiner, „Objekttechnologie": OO-Datenbanken, OO-Analyse, OO-Entwurf, …

Was ist ein Objekt?

DEF.: Ein Objekt ist ein Teil unserer Software: Eine (konkrete) Maschine, die durch die Operationen, die man auf ihr ausführen kann, charakterisiert ist.

DEF.: Ein Objekt ist eine Softwaremaschine, die es Programmen ermöglicht, eine Menge von Daten abzufragen und zu modifizieren.

Wir unterscheiden drei Objektarten:

  • Physische Objekte: repräsentieren materielle Objekte der modellierten Welt/Systems (z.B.: Hamster, Territorium, Korn, Kachel)
  • Abstrakte Objekte: repräsentieren abstrakte Dinge aus der modellierten Welt/System (z.B.: zurückgelegter Weg des Hamsters)
  • Softwareobjekte: repräsentieren reine Softwareobjekte (z.B.: Datenstrukturen wie Arrays oder Listen)

Eigenschaften von Objekten

  1. Ein Objekt verwaltet Daten , die im Speicher liegen
  2. Ein Objekt ist eine Maschine, die Operationen anbietet
  3. Ein Objekt hat eine Identität

Objekterzeugung

Basisprozess:

  • Allokiere neues Objekt im Haldenspeicher (Heap)
  • Initialisiere das Objekt mittels Erzeugungsprozedur
  • Verbinde Entität oder Variable mit neuem Objekt

Beispiel: Konstruktoren Location location = paule.getLocation(); Direction direction = paule.getDirection(); Territory territory = game.getTerritory(); Lokale Objektreferenzen = Zuweisung paula = new Hamster(); Objektvariable (Entität) = Zuweisung (noch nicht initialisiert) new erzeugt ein neues Objekt, d.h. einen neuen Hamster und gibt Objektreferenz auf dieses Objekt zurück. paula.init(territory, location, direction, 1); Explizite init von paula durch Operation mit Werten von paule

Zuweisung

Durch Zuweisungsinstruktionen können die (Laufzeit-) Werte von Variablen verändert werden.

Effekt (Pragmatik) einer Zuweisung:

Die Ausführung einer Zuweisungsinstruktion ziel = quelle; besteht aus:

Bestimmen des Werts des Ausdrucks “quelle”. Dafür zu sorgen, dass die Variable “ziel” von nun an für diesen Wert steht. Außerdem, Voraussetzung ist, dass der Typ von quelle kompatibel mit dem Typ von ziel ist (identisch oder durch Konvertierung)

Entität

Ist eines der folgenden:

  • Attribut (aka Objektvariable)
  • Klassenvariable
  • Lokale Variable einer Operation
  • Formaler Parameter einer Operation
  • this, als Objektreferenz auf das aktuelle Objekt

Variablen sind innerhalb des sie deklarierenden Blocks sichtbar!

Konstanten: Alle Variablen sollten final deklariert werden, außer es gibt gute Gründe dafür, dass mehrere Zuweisungen notwendig sind.

Erzeugungsprozeduren

Wenn Objekt nicht initialized, dann Operation nicht anwendbar. Falls Operationen einer Klasse die gleiche Vorbedingung besitzen, kann das auch als Invariante modelliert werden. Ansonsten Möglichkeit bereitstellen, um den Initialisierungscode auszuführen, während das Objekt erzeugt wird, und seine Nutzung erzwingen. -> Erzeugungsprozeduren aka Konstruktoren! Komfort: Initialisieren bei Erzeugung! Korrektheit: Invariante von Beginn an gewährleistet!

Prinzip der Erzeugung

Falls eine Klasse eine nicht-triviale Invariante besitzt, muss sie einen oder mehrere Konstruktoren anbieten, deren Zweck es ist sicherzustellen, dass jede Instanz, nach der Ausführung des Konstruktors, die Invariante erfüllt.

Operationen

Operationen können von Programmen auf bestimmte, eindeutig identifizierte Objekte angewendet werden und lösen ein eindeutig definiertes Verhalten dieser aus.

  • Eine Operation, die auf ein Objekt lesend zugreift, ist eine Abfrage
  • Eine Operation, die ein Objekt modifizieren kann, ist ein Kommando

Operationen eines Objektes fallen in unterschiedliche Kategorien:

  1. Abfragen
  2. Kommandos
  3. Erzeugungsprozeduren

Abfragen

Ziel: Die Eigenschaften von Objekten abfragen. Dabei sind Objekte idempotent, d.h. die Abfrage ändert nicht den Zustand des Objekts oder anderer Beteiligter Objekte.

Abfragen machen nichts Sichtbares, sondern liefern einen Wert mit dem weitergearbeitet werden kann.

Form der Deklaration von Abfragen: RETURN_TYPE operationsname()

Kommandos

Ziel: Den Zustand eines oder mehrerer Objekte ändern.

Das Prinzip der Trennung von Kommandos und Abfragen:

Eine Frage zu stellen, soll nie die Antwort verändern. Jede Verletzung dieser Regel macht Software schwerer verständlich und wartbar.

Argumente

Operationen können Argumente haben. In gut geschriebener OO-Software haben Operationen meist kein oder ein Argument, nicht mehr!

Daher: Jede Operation wird auf einem Objekt ausgeführt und kann aber muss nicht Argumente verarbeiten Bsp.: console.printf(" ARGUMENT"), territory.putWall( 1, 1 )

Ausdrücke

Ein Ausdruck ist ein Programmelement, welches für eine Menge von möglicher Laufzeitwerte steht.

Im Quellcode: Eine Instruktion steht für eine elementare Operation, die während der Programmausführung ausgeführt wird. Ein Ausdruck steht für einen (Laufzeit-)Wert, der von der Instruktion zur Ausführung benutzt wird.

API – Schnittstelle

DEF.: Eine Schnittstelle einer Menge von Softwaremechanismen ist die Beschreibung der Techniken, die es Kunden ermöglichen, diese Mechanismen zu nutzen.

Benutzeroberfläche: wenn Kunden menschlich sind

  • GUI
  • Textschnittstelle, Befehlszeilenschnittstelle

Programmschnittstelle: Kunden sind andere Softwaresysteme

  • API: Programmierschnittstelle (oder Anwendungsschnittstelle)

Der Entwerfer eines jeden Objekts muss klar spezifizieren, welche Operationen oder Eigenschaften für den Nutzer des Objekts öffentlich (public) verfügbar sind, und welche nur intern (geheim), sprich für den Implementierer des Objekts verfügbar sind.

Die Programmiersprache muss sicherstellen, das öffentliche Nutzer nur öffentliche Operationen und Eigenschaftenzugreifen können.

Ein Kunde (Aufrufer, Nutzer, Client) eines Softwaremechanismuses ist ein System jeglicher Art, - wie z.B. ein Softwareelement, ein Nicht-Softwaresystem oder ein menschlicher Benutzer – der dieses System verwendet.

Syntax, Semantik, Pragmatik

Syntax ≈ Gültige Struktur von Sätzen:

  • Art und Weise, Programme aufzuschreiben, die Gültigkeit wird vom Compiler geprüft

Semantik ≈ Bedeutung gültiger Sätze:

  • Erwarteter Effekt des Programms in seiner Umgebung, der generierte Maschinencode repräsentiert diese Bedeutung

Pragmatik ≈ Auswirkung gültiger Sätze:

  • Auswirkung auf die reale Welt, meist nicht Teil der Informatik, sondern der Anwendungsdomäne

Die Syntax eines Programms bestimmt die Struktur und die Form des Quellcodetexts. Die Semantik eines Programms ist die Menge der Eigenschaften seiner potenziellen Ausführungen.

Lexikale

Die elementaren Bestandteile des Programmtexts sind die Lexikale (Token):

  • Terminale
    • Bezeichner: Namen; vom Programmierer gewählt, z.B., paule oder display
    • Konstanten: Selbsterklärende Werte, z.B., 34
  • Schlüsselwörter, z.B. class
  • Spezialsymbole: z.B. Punkt, “.” genutzt in Operationsaufrufen

Lexikale definieren die lexikalische Struktur der Sprache

Regeln

  • Semantische Regeln definieren, was der Effekt von Programmen ist, die die syntaktischen Regeln erfüllen.
  • Statische Semantikregeln definieren Gültigkeitsregeln, die nicht in der Syntax erfasst wurden. (z.B.: Worte verdreht.)
  • Syntaktische Regeln definieren, wie man eine AST Struktur aus den gültigen Lexikalen ableiten kann.
  • Lexikalische Regeln definieren, wie eine Zeichenkette in Lexikale zerlegt wird.

Bsp.: Lexikalische Regeln für Bezeichner: Mindestens ein Zeichen (Buchstabe, Unterstrich, Dollarzeichen), weiter Zeichen dürfen Buchstaben, Zahlen, Unterstrich, Dollarzeichen

AST - Abstrakter Syntaxbaum

  • Zeigt die syntaktische Struktur auf
  • Nur Exemplare: keine Schlüsselwörter oder andere Trennzeichen (daher abstrakt)

Die Syntax einer Programmiersprache ist durch die Menge der Konstrukte (z.B. Klasse, Operation, Operationsaufruf, …) und die Struktur diese Konstrukte (z.B. Operationsdeklarationen sind Teil der Klassendefinition) definiert

  • Wurzel: repräsentiert das Gesamtexemplar
  • Innere Knoten (Nichtterminale): repräsentieren Unterstrukturen, die wiederrum Exemplare enthalten
  • Blätter (Terminale): repräsentieren Exemplare ohne weitere Untergliederung

Klassen

Jedes Objekt gehört zu einer bestimmten Klasse, die die anwendbaren Operationen (Kommandos oder Abfragen) definiert.

DEF.: Eine Klasse ist die Spezifikation (man sagt auch oft der Bauplan) für eine Reihe von möglichen Laufzeitobjekten, auf denen die gleichen Operationen ausgeführt werden können.

Eine Klasse repräsentiert eine Kategorie von Dingen. Ein Objekt repräsentiert eine spezielle Instanz dieser Dinge.

Klassen sind Schemata oder Vorlagen, Objekte sind die Instanzen oder Vorkommen ihrer Klasse.

Ist Objekt O durch Klasse c definiert, dann:

  • Nennt man O eine Instanz von C
  • Nennt man C die erzeugende Klasse von O

Klassen existieren (strenggenommen) nur im Quellcode.

Objekte existieren nur während der Ausführung.

Verträge

Nutzung von Verträgen:

  • Verträge zum Debuggen
  • Verträge zur Schnittstellendokumentation

DEF.: Ein Vertrag ist eine Menge von semantischen Bedingungen, die die Nutzungseigenschaften einer Klasse oder einer Operation charakterisieren.

Es gibt drei Vertragsarten: (Notation: /*@ … @*/)

  • Vorbedingungen
    • Ist eine Bedingung, die eine Operation jedem ihrer Aufrufer für einen erfolgreichen Aufruf auferlegt (requires frontIsClear(); )
    • Ein Aufrufer, der eine Operation aufruft, muss sicherstellen, dass die Vorbedingung der Operation vor dem Aufruf erfüllt ist.
  • Nachbedingungen
    • Ist eine Bedingung, die einer Operation für eine korrekte Realisierung durch den Anbieter auferlegt ist (ensures getDirection() == Direction.West)
    • \old: Schlüsselwort, um auf den Wert einer Abfrage zuzugreifen, bevor die Operation ausgeführt wurde
    • Eine Operationsrealisierung muss sicherstellen, dass, wenn die Vorbedingung der Operation zu Beginn der Ausführung erfüllt ist, die Operationsausführung an deren Ende die Nachbedingung erfüllt.
  • Klasseninvarianten
    • Sind Bedingungen, die zwischen beliebigen Operationsausführungen auf den Objekten der Klasse gelten müssen (invariant)

Stilregeln für Verträge:

  • Zusammenfassung in einem kurzen Satz
  • Detaillierte Beschreibung
  • Liste der anwendbaren Tags mit Spezifikationen/Bedeutung
  • Bei Operationen: Vor- und Nachbedingungen (min. nat. Sprache, besser in nat. und JML)
  • Bei Klassen: Klasseninvarianten (min. nat. Sprache, besser in nat. und JML)

Logik

Eine Bedingung wird ausgedrückt als boolscher Ausdruck. Dieser besteht aus:

  • Boolschen Variablen (Bezeichner für boolsche Werte)
  • Boolschen Operatoren (nicht, oder, und, =, $\Rightarrow$)

Und repräsentiert mögliche:

  • Boolsche Werte (true, false)

Und und Oder

Prinzip der Disjunktion: Eine oder-Disjunktion hat den Wert wahr, außer beide Operanden haben den Wert falsch.

Oder-Operator ist nicht-ausschließend und kommutativ.

Und-Operator ist kommutativ. Konjunktionsprinzip: Eine und-Konjunktion hat den Wert falsch außer beideOperanden haben den Wert wahr.

Dualität von und und oder : Die Eigenschaften des einen Operators ergeben die Eigenschaften des anderen Operators (durch Negierung und Tauschen von wahr und falsch)

Wahrheitsbelegung und Wahrheitstabellen

Die Wahrheitsbelegung einer Menge von Variablen ergibt sich durch die Wahl der Werte für jede Variable, sprich jede Variable hat entweder den Wart wahr oder falsch. Eine Wahrheitsbelegung erfüllt einen Ausdruck, wenn der Wert des Ausdrucks wahr ist.

Tautologien

Tautologie: ein boolscher Ausdruck hat den Wert Wahr für jede mögliche Wahrheitsbelegung. Bsp.:

  • a oder (nicht a)
  • nicht (a und (nicht a))
  • (a und b) oder ((nicht a) oder (nicht b))

Widersprüche

Widerspruch: Ein boolscher Ausdruck, der für alle möglichen Wahrheitsbelegungen den Wert falsch hat. Bsp.: a and (not a)

Erfüllbarkeit

Erfüllbar: für mindestens eine Wahrheitsbelegung ergibt der Ausdruck den Wert wahr.

Jede Tautologie ist erfüllbar. Kein Widerspruch ist erfüllbar.

Gleichheit (=)

= ist kommutativ (a=b ist dasselbe wie b=a) und reflexiv (a=a ist Tautologie für beliebige a)

De Morgansche Gesetze

De Morgansche Gesetze: Tautologien

  • (nicht (a oder b)) = ((nicht a) und (nicht b))
  • (nicht (a und b)) = ((nicht a) oder (nicht b))

Mehr Tautologien (Distributivität):

  • (a und (b oder c)) = ((a und b) oder (a und c))
  • (a oder (b und c)) = ((a oder b) und (a oder c))

Syntaxkonventionen: Binden von Operatoren

Reihenfolge nach Bindung (beginnend mit stärkster Bindung):

  • nicht, und, oder, impliziert, =

und und oder sind assoziativ:

  • a und (b und c) = (a und b) und c
  • a oder (b oder c) = (a oder b) oder c

Implikation

a $\Rightarrow$ b, hat für beliebige a und b den Wert (nicht a) oder b Im Ausdruck a $\Rightarrow$ b ist a die Voraussetzung, b die Folgerung.

Implikationsprinzip:

  • Eine Implikation hat den Wert wahr außer die Voraussetzung hat den Wert wahr und Folgerung hat den Wert falsch,
  • Insbesondere, immer den Wert wahr, wenn die Voraussetzung den Wert falsch hat.

Immer wenn a Falsch ist, ist a $\Rightarrow$ b Wahr, unabhängig von b:

Fälle in denen a Falsch ist, sagen uns nichts über den Wahrheitswert der Folgerung aus.

Implikationen umkehren

Es ist nicht grundsätzlich wahr, dass: $a \Rightarrow b = (\neg a)\Rightarrow(\neg b)$

Korrekt: $a \Rightarrow b = (\neg b) \Rightarrow (\neg a)$

Semistrikte boolsche Operatoren

a und dann b: hat denselben Wert wie a und b, wenn a und b definiert, und ist immer Falsch, wenn a den Wert Falsch hat.

a oder sonst b: hat denselben Wert wie a oder b, wenn a und b definiert, und ist immer Wahr, wenn a den Wert Wahr hat.

Semistrikte Operatoren erlauben uns die Auswertungsreihenfolge eines Ausdrucks zu definieren. Und dann: nur wenn eine Bedingung Sinn macht, falls eine andere wahr ist. oder sonst: wenn eine Bedingung nur Sinn macht, falls eine andere falsch ist. gewöhnliche boolsche Operatoren: wenn garantiert werden kann, dass beide Operanden definiert sind.

Referenzen

DEF.: Ein Bezeichner ist ein von Programmieren gewählter Name, der bestimmte Elemente des Programms repräsentiert. Ein Bezeichner, der einen Laufzeitwert bezeichnet, nennt man, oder eine Variable, falls sie ihren Wert ändern kann. Variablen, die einen Wert pro Objekt haben, werden Objektvariablen genannt.

Während der Ausführung kann eine Entität an ein Objekt gebunden werden.

Spezielle Referenzen: null

Zu beginn ist in Java die Objektreferenz paula, nicht an ein Objekt gebunden: der Wert ist eine sogenannte null _-Referenz.

Optional

Anstelle Objekte nicht zu binden, Verwendung von Optional. Beispiel: Optional<Person> spouse; $\rightarrow$ Optionale Referenz auf ein Person-Objekt.

Zustände einer Referenz

Während der Ausführung (Laufzeit) ist eine Referenz entweder:

  • Gebunden an ein bestimmtes Objekt
  • Null (ungebunden)

Wir vermeiden die Nutzung von Objektreferenzen mit null-Werten grundsätzlich! Objektreferenzen müssen immer direkt mit einer gültigen Referenz initialisiert werden!

this

Manchmal benötigen wir eine Referenz auf das aktuelle Objekt: Beispiel in der Hamster-Klasse:

Hamster getCurrentHamster() {
	return this;
 }

In den Operationen und Erzeugungsprozeduren einer Klasse, zeigt die this-Referenz immer auf das aktuelle Objekt.

this ist vom Typ der enthaltenden Klasse: In der Klasse Hamster, ist this vom Typ Hamster. In der Klasse Territory, ist this vom Typ Territory.

Invariante von this: this != null

Algorithmus

Ein Algorithmus ist die Spezifikation eines Prozesses, der von einem Computer ausgeführt werden soll.

Eigenschaften:

  1. Definiert die Daten, auf denen der Algo arbeitet
  2. Jeder Basisschritt stammt aus einer Grundmenge klar definierter Aktionen
  3. Beschreibt die Ordnung dieser Schritte
  4. Eigenschaften 2&3 basieren auf präzise definierte Konventionen, die für eine automatische Verarbeitung geeignet sind.
  5. Es ist garantiert, dass der Algo nach einer endlichen Anzahl an Schritten terminiert.

Kontrollflussstrukturen

DEF.: Kontrollflussstrukturen sind Programmkonstrukte, die die Reihung der Basisschritte beschreiben bzw. bestimmen. Drei Fundamentale Strukturen:

  • Sequenzen Mehrere Operationen hintereinander
  • Verzweigungen
    • If / Else:
if (Boolscher Ausdruck) {
	Instruktionen
	} else if {
		Andere Instruktionen
		} else {
			...
}
  • Switch:
switch (intGrade) {
case 1: …
case 2: …
default: ...
}
  • Schleifen
for (int i=0; i < 3; i++) {
	paule.move();
}

do {
	paule.move();
} while (paule.frontIsClear());

while (paule.frontIsClear()) {
	paule.move();
}

Schleifeninvarianten

Eine Schleifeninvariante ist eine Bedingung mit folgenden Eigenschaften:

  • Ist erfüllt (d.h. Wahr) nach der Schleifeninitialisierung
  • Wird bewahrt von jedem Schleifendurchlauf, der ausgeführt wird, weil die Schleifenabbruchbedingung nicht erfüllt ist.
  • Soll erfassen, wie weit die Arbeit der Schleife vorangeschritten ist.

Ist eine Ganzzahl, für die gilt:

  • Positiv nach der Schleifeninit.
  • Wird für jede Schleifeniteration verringert und bleibt positiv, solange Abbruchbedingung nicht erfüllt, ist

Gleichförmiger Zugriff (UAP)

Prinzip: Für den Aufrufer sollte es keine Rolle spielen, ob ein Ergebnis durch Berechnung oder Speicherzugriff bestimmt wird.

Operationsimplementierung umgehen mit Vor-&Nachbedingung

Defensives Programmieren: Explizite Prüfung bei der Bedingung. Bei Nichterfüllung wird eine Exception geworfen und der Zustand nicht geändert. Für alle public Operationen verwenden, die durch dritte zugänglich sind.

Offensives Programmieren: Bedingungen werden mittels assert zur Entwicklungszeit geprüft und zur Produktionszeit deaktiviert. Für alle internen Operationen verwenden.

Vererbung

Klassen in der echten Welt besitzen eine Vielzahl an Beziehungen. Eine wichtige Beziehung ist die „ist ein(e)" Beziehung. Bsp. Labrador is Dog, is Mammal, is Anmimal.

DEF.: Vererbung erlaubt es Programmierern, Klassen zu erstellen, die auf existierenden Klassen basieren, eine neue Realisierung zu spezifizieren, während das Verhalten beibehalten wird, um Code wiederzuverwenden und um die ursprüngliche Software durch öffentliche Klassen und Schnittstellen zu erweitern.

Für eine Klasse A:

  • Die Nachkommen von A sind A selbst und (rekursiv) die Nachkommen von As Erben
  • Echte Nachkommen schließen A selbst aus

Umgekehrte Notation:

  • Vorfahren
  • Echte Vorfahren

Präzisierte Vorstellung von Instanzen:

  • Direkte Instanzen von A
  • Instanzen von A: die direkten Instanzen von A und dessen Nachkommen

Grundlagen der Vererbung

Prinzip: Beschreibe eine neue Klasse als Erweiterung oder Spezialisierung von einer existierenden Klasse.

Falls „Klasse B von Klasse A erbt":

Modul Blickpunkt: Alle angebotenen Klassenelemente (Operationen, Konstruktoren, Attribute, …) von A sind auch in B verfügbar. Aka Code Sharing. Stilregel: Komposition vor Vererbung.

Typ Blickpunkt: Immer, wenn eine Instanz (Objekt) einer Klasse A benötigt wird, wird auch eine Instanz (Objekt von B akzeptiert.)

DEF.: Arten von Operationen/Attributen „Operationen/Attribute einer Klasse" sind eines von beidem:

Eine geerbte Operation/Attribut, falls sie eine Operation/Attribut von einem der Eltern der Klasse ist.

Eine unmittelbare Operation/Attribut, falls sie in der Klasse deklariert und nichtgeerbt wurde. In diesem Falls sagt man, dass die Klasse diese Operation/Attribut eingeführt hat.

Abstraktion

Manche Klassen können keine Implementierung ihrer Operation anbieten. $\rightarrow$ also Operation deklarieren, die nur aus der Spezifikation besteht. Das kann die Signatur, Vor-/ Nachbedingung etc. umfassen. Schlüsselwort: abstract

Abstrakte Operationen besitzen keinen Operationsrumpf, und dürfen nicht private sein. Jede Klasse mit abstrakten Operationen, kann nicht mit new instanziiert werden und muss ebenfalls als abstrakt deklariert werden. Müssen in Subklassen implementiert werden.

Überschreiben

@override, informiert den Compiler, dass diese Operation eine spezifizierte, geerbte Operation implementiert. Wenn eine Operation als final deklariert wird, kann sie von keiner Kindklasse überschrieben werden. Wenn eine Klasse als final deklariert wird, kann es keine Nachkommen dieser Klasse geben.

Stilregel: Alle Operationen, die nicht absichtlich dafür erstellt wurden, um in Subklassen überschrieben zu werden, sollten final sein.

Polymorphie

Wir wollen Algorithmen schreiben, die auf allgemeine Typen von Objekten anwendbar sind, aber trotzdem die Besonderheiten von spezialisierten Typen einbezieht.

DEF.: Polymorphie

Eine Bindung (auch bei Zuweisung oder Argumentübergabe) ist polymorph, wenn Zielvariable und Quellausdruck von unterschiedlichem Typ sind. Eine Entität oder ein Ausdruck sind polymorph, wenn sie zur Laufzeit – als Resultat einer polymorphen Bindung – an Objekte unterschiedlichen Typs gebunden werden. Polymorphie ist die Existenz dieser Möglichkeiten in einer Programmiersprache.

Statische und Dynamische Typen

Der statische Typ einer Entität ist der Typ, der in ihrer Deklaration in dem zugehörigen Klassentext steht. Bsp.: Animal pet1 $\rightarrow$ statischer Typ von pet1 ist Animal.

Wenn der Wert der Entität, während einer bestimmten Ausführung, an ein Objekt gebunden wird, ist der Typ dieses (Laufzeit-) Objekts der dynamische Typ der Entität zu dieser Zeit. Bsp.: pet1 = paule; $\rightarrow$ Dynamischer Typ von pet1 ist Hamster nach dieser Zuweisung

Der dynamische Typ einer Entität muss immer ihrem statischen Typ konform sein (d.h. ein Nachfahre)

Statische Typisierung:

  • Typsichere Aufrufe: Ein Operationsaufruf x.f() bei dem das an x gebundene Objekt eine Operation entsprechend zu f hat.
  • Statische Typprüfung: Ein programmverarbeitendes Werkzeug (Compiler), welches für jedes von ihm akzeptierte Programm garantiert, dass jeder Aufruf in jeder beliebigen Ausführung typ-sicher ist.
  • Statisch typisierte Sprachen: Eine Programmiersprache, für die es möglich ist, eine statische Typprüfung zu schreiben.

Dynamisches Binden:

Jede Ausführung eines Operationsaufrufs wird die Version der Operation verwendet, die dem dynamischen Typ des Zielobjekts am besten angepasst (die spezialisierteste) ist.

Überladen vs. Überschreiben von Operationen

Häufig hätten wir gerne mehrere Operationen, die alles dasselbe Verhalten realisieren. Lösung: Überladen von Operationen Eigenschaften: Operationen oder Konstruktoren, mit dem gleichen Namen, aber Argumenten von unterschiedlicher Anzahl oder Typ, werden vom Compiler als unterschiedliche Operationen behandelt.

Vererbung und Verkettung von Konstruktoren

Konstruktoren werden grundsätzlich nicht an die Kindklasse vererbt. Sie können jedoch aufgerufen werden. Eigenschaften: Verwende super(arg1, arg2, …, argN), um einen Konstruktoren der Elternklassen aufzurufen. Super-Aufrufe müssen der erste Aufruf im Konstruktor sein!!! Falls kein Standardkonstruktor in der Elternklasse vorhanden ist, muss der Aufruf explizit erfolgen. Super-Aufrufe können überladene Aufrufe sein.

Super-Zugriff

Code aus überschriebenen Operationen wiederverwenden: super.method() Auf sichtbare Attribute kann auch über super.attribute() zugegriffen werden, falls die Kindklasse das gleiche Attribut hat (was aus Stilgründen vermieden werden sollte).

Eine Collection (Sammlung) – manchmal auch Container (Behälter) genannt – ist einfach ein Objekt, dass andere Objekte in einem einzelnen Objekt gruppiert. Collections werden zum Speichern, Abfragen, Manipulieren und Übertragen von aggregierten Daten benutzt.

Collection-Objekte bieten einige grundlegende Operationen auf einem Container:

  • Iteration (or „traversal"): Anwenden einer bestimmten Operation auf jedes Element

Rekursion

Eine Definition für ein Konzept ist rekursiv, wenn sie eine Instanz des Konzepts selbst betrifft.

Rekursive Operationen

Direkte Rekursion: Operationsrumpf beinhaltet einen Aufruf der Operation selbst. Indirekte Rekursion: Operation r ruft s auf. S ruft r auf. Oder r1 ruft r2…ruft rN ruft r1 auf.

Nützliche rekursive Definition

Sollte sicherstellen, dass:

  • R1 Es gibt mindestens einen nicht rekursiven Zweig
  • R2 Jeder rekursive Zweig trifft in einem Kontext auf, der ich vom Original unterscheidet.
  • R3 Für jeden rekursiven Zweig bringt der Kontextwechsel (R2) ihn näher an mindestens einen der nicht rekursiven Fälle (R1)

Fakultät berechnen

0! = 1 n! = n * (n-1)! requires num $\geq$ 0 ensures \ result = num * (num-1) *…* 1

	if (num \<=1) {
		//Abbruch der Rekursion, kennen bereits die Lösung//

Rekursionsvariante – Korrektheit der Rekursion

Jede rekursive Operation sollte eine Rekursionsvariante verwenden, eine ganzzahlige Menge, die einem Aufruf zugeordnet ist, so dass:

  • Die Variante ist immer $\geq$ 0
  • Wenn eine Operationsausführung mit dem Variantenwert v beginnt, erfüllt der Wert v’ für jeden rekursiven Aufruf 0 $\leq$ v’ $<$ v

Eigenschaft eines Systems, in dem verschiedene Aktivitäten gleichzeitig ausgeführt werden können.

Die Aktivitäten können nur gleichzeitig ausgeführt werden, wenn zwischen ihnen keine kausale Abhängigkeit besteht.

      • MIPS

    • Echte Parallelität

  • Multiprozessoren-Systeme
    • mehrere Prozessoren, eventuell mit mehreren Kernen
    • lokaler Speicher pro Prozessor
    • häufig globaler Speicher, gemeinsam genutzt von allen Prozessoren
    • Echte Parallelität

  • Echte Parallelität
    • zwischen den Knoten
    • Zwischen Prozessoren eines Knotens
    • Zwischen Kernen eines Prozessors

Sequentielle und Nebenläufige Programme

Sequentielles Programm:

Alle Instruktionen des Programms werden sequentiell ausgeführt.

Nebenläufiges Programm / Concurrent Programm

Besteht aus einer Menge sequentiller Programme, die parallel ausgeführt werden können.

Der Begriff

  • “parallel” bezeichnet typischerweise echte Prarallelität
  • “nebenläufig” bezeichnet mögliche Parallelität, d.h echte Parallelität und Quasiparallelität

Vorteile der Nebenläufigkeit

  • Ausnutzung mehrerer Prozessoren
    • sequentielle Programm kann bei 100 Prozessoren max. 1% der Rechenkapazität nutzen
    • Nebenläufiges Programm kann potentiell gesamte Kapazität nutzen, d.h. verbesserter Durchsatz
    • Beachte: Selbst bei einem Einprozessor-System, lässt sich durch Quasi-Parallelität der Durchsatz deutlich erhöhen
  • Einfache Modellierung
    • Reale Prozesse bestehen häufig aus nebenläufigen Aktivitäten
    • “Natürliche” Modellierung des Prozesses durch ein nebenläufiges Programm, in dem die einzelne Aktivität durch sequentielle Programme realisiert werden.

Herausforderungen der Nebenläufigkeit

Nebenläufige Programme erfordern Kommunikation und Synchronisation zwischen Prozessen.

  • Fehlerfreie und effiziente Synchronisation und Kommunikation sind schwierig!
  • Probleme sind häufig situations- und/oder zeitabhängig
    • schwierig zu reproduzieren, diagnostizieren, korriegieren

Beispiele:

  • Wettlaufsituation (Race Conditions)
    • wenn das Ergebnis einer Berechnung vom zeitlichen Verhalten einzelner Operationen abhängt
  • Verklemmung (Deadlocks)
    • wenn sich eine Menge von Prozessen zyklisch gegenseitig blockieren

Data Staging Area

The Data Staging Area (DSA) is a storage area and a set of processes that clean, transform, combine, deduplicate, household, archive, and prepare source data for use in the data warehouse.

  • temporarily stored in staging area, before loaded in the warehouse
  • transformations are performed on DSA
  • Preprocessing does not influence data sources or data warehouse
  • DSA is central repository for ETL (Extraction, Transformation, Load)

Extraction

Transfers data from data source into the data staging area.

  • Multiple extract types
    • periodic
    • started by admin/user
    • event-driven
    • immediate after changes in data sources
  • Method depends on used monitoring strategy
  • extracted subset of data sources and schedule of the extraction, depends on analysis

Transformation

Convert the data into something representable to the users and valuable to the business.

  • Structure and Content
    • denormalization, normalization (First Normal Form, Second NF, …)
    • data type conversion (int -> float)
    • calculation, aggregation
    • standardization of strings and date values (currency, data formats)
    • conversion of measures (liters -> gallons)
    • cleansing (missing, wrong, inconsistent)
      • wrong/missing postcode
      • use correct data and build from there

Load

Transfer data from the data staging area into the core data warehouse.

  • Data in core data warehouse is only added, rarely replaced.
    • History of changes is stored
  • mainly based on bulk load tools of the DBMS
  • Offline vs. Online Load
    • Online: DW is still in use while uploading.
  • Parallel load may be required
    • controlling relational dependencies(PK,FK, …)

Core Data Warehouse

Integrated data as basis for all kinds of analytic purposes.

  • Characteristics
    • integrated view on all data sources (schema and data)
    • integrates cleanses data from the DSA
    • current and history data, granularity (lowest relevant granularity)
    • no focus on specific type of analysis (in contrast to data mart)
    • source for the data marts
    • single point of truth!
